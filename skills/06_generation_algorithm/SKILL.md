# Алгоритм генерации тестовых сценариев

## Шаг 1. Анализ требований

Проанализировать требования. Выделить:
- акторов
- предусловия
- основной сценарий
- альтернативные сценарии
- критерии приёмки

## Шаг 2. Разрешение неоднозначностей

Проверить наличие неоднозначностей. Если обнаружены:
- не генерировать тесты
- сформировать вопросы

## Шаг 3. Применение техник тест-дизайна

Применить техники тест-дизайна (см. `03_test_design_techniques`).

## Шаг 4. Группировка требований по бизнес-идеям

Группировать требования по бизнес-идеям, НЕ по строчкам требований.

**Главное правило:** один TC на бизнес-поток, а не на строчку требований.

### Техника: «Единый TC с ветвлениями»

Когда несколько вариантов имеют **одинаковый тип проверки** (фильтр: выбран → применён; фильтр: снят → сброшен; каждый предмет в группе; каждая форма обучения), объединяем в один TC:

```
TC-XXX – Название фильтра (применение и сброс)

Шаги:
1. Нажать кнопку фильтра.
2. Нажать эту же кнопку повторно.

Ожидаемый результат:
1. Отображён отфильтрованный список (соответствует критерию).
2. Фильтр сброшен, отображён полный список.
```

Каждый «вариант» = отдельный шаг + отдельный пункт ожидаемого результата (строго 1:1, как требует шаблон `02_test_case_template`).

**Пример НЕгруппировки (ПЛОХО):**
- TC-001: Фильтр по бюджетным местам — выбран
- TC-002: Фильтр по бюджетным местам — снят
- TC-003: Фильтр по общежитию — выбран
- TC-004: Фильтр по общежитию — снят

**Пример группировки (ХОРОШО):**
- TC-001: Фильтр «Бюджетные места» — применение и сброс
- TC-002: Фильтр «Общежитие» — применение и сброс

### Правила группировки

- Один UI-элемент в одном контексте → happy path и связанные negative кейсы объединяются в единый TC с ветвлениями
- Альтернативные контексты → отдельные тесты только если поведение РАЗЛИЧАЕТСЯ
- Логика бюджета → ОБЪЕДИНИТЬ в единый TC с ветвлениями (каждая комбинация = шаг + ожидаемый результат)
- Группы однотипных элементов (предметы, отрасли, формы обучения) → единый TC с ветвлениями по каждому варианту

### Контроль фрагментации

Количество TC определяется пропорционально объёму требований. Если количество TC превышает количество ключевых требований в 2.5 раза → пересмотри группировку.

**Не задавай жёстких цифровых лимитов** — ориентируйся на соотношение TC : ключевые требования. Жёсткие числа (20-25, 30+, 4-5) устарели и ведут к искусственному зауживанию или раздуванию.

## Шаг 5. Генерация тестовых сценариев

Сгенерировать тестовые сценарии (следя за группировкой из Шага 4).

## Шаг 6. Самопроверка (СТРОГО — НЕ выводить результат без прохождения)

Выполнить ВСЕ проверки ВНУТРЕННЕ, до вывода результата:

### 6.1 Контроль фрагментации

Посчитать соотношение: количество тестов : количество ключевых требований.

Если ratio > 2.5:1 → тесты фрагментированы, сгруппируй.

**Правило группировки:**
- Все проверки одного элемента интерфейса в одном контексте → один тест (счастливый путь + негативный путь в одном TC с ветвлениями)
- Альтернативные контексты (каталог vs страница специальности) → отдельные тесты только если поведение РАЗЛИЧАЕТСЯ
- Бюджетные поля (балл + места + рекомендация при 0/NULL + скрытие без бюджета) → ОБЪЕДИНИТЬ в единый TC с ветвлениями

**Пример фрагментации (ПЛОХО):**
- TC-001: Город показан с данными
- TC-002: Город скрыт при NULL
- TC-003: Балл показан в каталоге
- TC-004: Балл показан на странице специальности
- TC-005: Рекомендация при балле = 0

**Пример группировки (ХОРОШО):**
- TC-001: Отображение карточки с городом (счастливый путь)
- TC-002: Город не отображён при NULL/пусто
- TC-003: Строка «Бюджет» — число при >0, рекомендация при 0/NULL, скрыта без бюджета

### 6.2 Повторная проверка атомарности

Каждый тест должен проверять ОДНУ бизнес-идею.

Бизнес-идея ≠ строчка из требований.

Бизнес-идея — это «пользователь видит корректную карточку» (объединяет: наименование, город, баллы, стоимость, счётчик).

Строчка из требований — это «Город: не выводить при NULL».

### 6.3 Контроль структуры 1:1

**Обязательная механическая проверка для КАЖДОГО TC перед выводом:**
- Посчитать N (шаги) и M (ожидания). Если N ≠ M — тест не выводить.
- Если M > N — объединить лишние пункты вложенными буллитами под соответствующий шаг.

Полное правило 1:1 и механическая проверка — см. `04_quality_and_best_practices` (раздел «Пошаговая проверка»). Это единый источник истины для всех правил 1:1 в нашей системе.

### 6.4 Контроль покрытия

Пройти по каждому критерию приёмки и убедиться, что хотя бы один тест его покрывает.

### 6.4 Контроль качества вывода

- Нет тривиальных тестов типа «кнопка отображена» (кроме критичных точек входа)
- Присутствуют вопросы для уточнения неоднозначностей
- Предположения явно маркированы

### 6.5 Проверка антипаттернов

Проверить по таблице антипаттернов:

| Антипаттерн | Признак | Решение |
|---|---|---|
| Полевой тест | «Поле X отображено/скрыто» | Объединить с контекстом |
| Дублирующий счастливый путь | Один и тот же счастливый путь для разных ролей | Объединить в один тест с перечислением ролей |
| Контекстный разброс | Один тест на каждый контекст для одного поля | Объединить, если поведение одинаковое |
| Zero-test | Тест только на значение = 0 для каждого поля | Объединить все zero/null edge-кейсы |
| Button-only | «Кнопка отображена» без действия | Объединить с навигацией/счастливым путём |

### 6.6 Дисциплина вывода

ВЫВОДИТЬ ТОЛЬКО итоговый список тест-кейсов.
- НЕ выводить черновики или промежуточные наборы
- НЕ перечислять «Исправленный вариант» в конце ответа
- НЕ выводить расчёты или промежуточные шаги группировки

## Шаг 7. Финальная проверка

Проверить:
- полноту покрытия (каждый критерий приёмки покрыт)
- отсутствие дублирования
- отсутствие галлюцинаций
- воспроизводимость

## Шаг 8. Возврат результата

Вернуть только итоговый набор тестовых сценариев.
