# Качество и лучшие практики QA

## Полнота покрытия

Все критерии приемки должны быть покрыты хотя бы одним тестовым сценарием.

## Соответствие критериям приемки

Тестовые сценарии должны проверять заявленную функциональность.

## Воспроизводимость

Тестовый сценарий должен быть воспроизводим специалистом, ранее не знакомым с проектом.

Не допускаются абстрактные формулировки:
- корректные данные
- вызвать ошибку
- инициировать сбой
- выполнить частичный отказ

Если тестовый сценарий требует моделирования специальных условий, способ их воспроизведения должен быть явно описан.

## Абстрактные значения

Не использовать абстрактные значения в требованиях:

Плохо:
- валидный email
- корректный пароль

Хорошо:
- email в формате user@example.com
- пароль длиной 5 символов

Использовать классы эквивалентности вместо фиксированных значений:

Email:
- валидный email
- email без символа «@»
- email без доменной части

Пароль:
- пароль минимальной длины
- пароль меньше минимальной длины
- пароль без специальных символов

Конкретные тестовые данные могут быть выбраны позже на этапе подготовки. Не привязывать тестовые сценарии к одному фиксированному набору данных.

Избегать неопределённых формулировок.

Плохо: «слабый пароль»
Хорошо: «пароль длиной 5 символов»

Не использовать внутренние детали реализации, если они отсутствуют в требованиях. Не создавать лишние проверки.

## Отсутствие дублирования

Не создавать несколько тестов, проверяющих одинаковую бизнес-логику.

## Отсутствие галлюцинаций

Запрещено добавлять отсутствующие в требованиях сущности и бизнес-правила.

## Атомарность

Каждый тестовый сценарий должен проверять ОДНУ бизнес-идею.

Не допускается объединение нескольких ветвей логики, условий или вариантов поведения в одном тестовом сценарии.

Плохо:
- Если пользователь администратор — отображается кнопка удаления
- Если пользователь менеджер — отображается кнопка редактирования

Характерный признак нарушения — использование конструкций:
- если … то …
- иначе …
- в случае …
- при наличии …

В таких случаях необходимо формировать отдельные тестовые сценарии:

TC-001 — Отображение кнопки удаления для администратора
TC-002 — Отображение кнопки редактирования для менеджера

Тест считается неатомарным, если содержит:
- несколько альтернативных путей выполнения
- условные конструкции: если …; иначе …; при наличии …; в случае …

Неатомарные тестовые сценарии должны быть разделены на несколько независимых тестов.

## Пошаговая проверка

Каждый шаг должен иметь собственный ожидаемый результат.

Критерий считается выполненным, если:
- каждому шагу соответствует один ожидаемый результат
- отсутствуют шаги без ожидаемых результатов
- отсутствуют ожидаемые результаты без соответствующего шага
