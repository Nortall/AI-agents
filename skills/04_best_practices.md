# QA Best Practices

Не использовать абстрактные значения:

Плохо:

- валидный email;
- корректный пароль.

Хорошо:

- email в формате user@example.com;
- пароль, удовлетворяющий требованиям.

Использовать классы эквивалентности вместо фиксированных значений.

Примеры:

Email:

- валидный email;
- email без символа "@";
- email без доменной части.

Пароль:

- пароль минимальной длины;
- пароль меньше минимальной длины;
- пароль без специальных символов.

Конкретные тестовые данные могут быть выбраны позднее на этапе подготовки тестовых данных.

Не привязывать тестовые сценарии к одному фиксированному набору данных.

Избегать неопределенных формулировок.

Плохо:

"слабый пароль"

Хорошо:

"пароль длиной 5 символов"

Не использовать внутренние детали реализации, если они отсутствуют в требованиях.

Не создавать лишние проверки.

Каждый тест должен проверять одну бизнес-идею.

Шаги должны быть понятны специалисту, незнакомому с проектом.

## Atomicity Principle

Каждый тестовый сценарий должен проверять только одну бизнес-идею.

Не допускается объединение нескольких ветвей логики, условий или вариантов поведения в рамках одного тестового сценария.

Плохо:

- Если пользователь администратор — отображается кнопка удаления.
- Если пользователь менеджер — отображается кнопка редактирования.

Характерный признак нарушения:

использование конструкций:

- если ... то ...;
- иначе ...;
- в случае ...;
- при наличии ...;

В таких случаях необходимо формировать отдельные тестовые сценарии.

Хорошо:

TC-001 — Отображение кнопки удаления для администратора.

TC-002 — Отображение кнопки редактирования для менеджера.
