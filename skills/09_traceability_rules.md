# Traceability Rules

## Purpose

Каждый тестовый сценарий должен быть явно связан с одним или несколькими требованиями.

Трассируемость используется для:

- контроля полноты покрытия требований;
- обнаружения галлюцинаций;
- выявления избыточных тестов;
- проверки соответствия тестовых сценариев бизнес-требованиям.

---

# Allowed Sources

Источником тестового сценария могут являться только:

- основной сценарий;
- альтернативный сценарий;
- критерий приемки;
- явно сформулированное бизнес-правило;
- нефункциональные требования (если они присутствуют).

Не допускается создание тестовых сценариев, не имеющих источника происхождения.

---

# Requirement → Test Cases

Для каждого требования необходимо определить связанные тестовые сценарии.

Пример:

A1 Email уже существует
→ TC-002

A2 Неверный формат email
→ TC-003

A3 Слабый пароль
→ TC-004

Если требование не связано ни с одним тестовым сценарием, то покрытие считается неполным.

---

# Test Case → Requirement

Для каждого тестового сценария необходимо определить требование-источник.

Пример:

TC-001
Источник:
Основной сценарий

TC-002
Источник:
A1

TC-003
Источник:
A2

TC-004
Источник:
A3

Если источник определить невозможно, тестовый сценарий считается подозрительным и должен быть исключен.

---

# Hallucination Detection

Признаки галлюцинации:

- тест не связан ни с одним требованием;
- тест использует несуществующие сущности;
- тест использует отсутствующие статусы;
- тест использует отсутствующие коды ошибок;
- тест использует неописанные бизнес-правила.

Пример.

Требования:

Пользователь регистрируется.

Сгенерированный тест:

Проверка статуса pending_activation.

Статус pending_activation отсутствует в требованиях.

Такой тест является галлюцинацией и должен быть удален.

---

# Coverage Verification

После генерации тестовых сценариев необходимо убедиться, что:

- основной сценарий покрыт тестами;
- альтернативные сценарии покрыты тестами;
- критерии приемки покрыты тестами;
- каждый тест имеет источник происхождения.

---

# Bidirectional Traceability Matrix

После генерации необходимо построить двунаправленную матрицу трассировки.

Требование → Тесты

Основной сценарий
→ TC-001

A1
→ TC-002

A2
→ TC-003

Критерий приемки AC-1
→ TC-001

---

Тест → Требование

TC-001
→ Основной сценарий
→ AC-1

TC-002
→ A1

TC-003
→ A2

---

# Quality Rules

НЕ допускается ситуация:

Одно требование → несколько тестов.

Например:

A2 Невалидный email

→ TC-003 Отсутствует "@"
→ TC-004 Отсутствует домен
→ TC-005 Отсутствует локальная часть

Решение: объединить проверки валидации в один тест-кейс

Не допускается ситуация:

Тестовый сценарий → отсутствие требования.

Такие тесты считаются галлюцинациями.

---

# Self-Check Algorithm

После завершения генерации выполнить проверку:

1. Для каждого требования найти связанные тесты.

Если тесты отсутствуют:

→ создать дополнительные тестовые сценарии.

2. Для каждого тестового сценария найти источник происхождения.

Если источник отсутствует:

→ удалить тестовый сценарий.

3. Убедиться, что все критерии приемки покрыты.

4. Вернуть только итоговый набор тестовых сценариев.

---

# Main Principle

Каждый тестовый сценарий должен отвечать на вопрос:

"Какое именно требование я проверяю?"

Если ответ на этот вопрос отсутствует, тестовый сценарий не должен присутствовать в итоговом наборе.
