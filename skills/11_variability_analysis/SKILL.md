# Анализ вариантов (Variability Analysis)

## Назначение

Перед генерацией тестовой документации проанализировать требования на наличие сущностей с множественными состояниями, вариантами выбора или условным поведением. Это позволяет выявить скрытые тестовые пространства, которые не очевидны при поверхностном чтении.

## Когда применять

Перед Steps 1–2 алгоритма `06_generation_algorithm`:
- В требованиях есть сущности с перечислимые значениями (статусы, роли, типы).
- Есть условная логика («если X, то Y»).
- Есть поля, которые могут принимать NULL/пусто/0.
- Есть связи между сущностями (зависимые данные).

## Процедура анализа

### Шаг 1. Выявление сущностей

Найти в требованиях бизнес-сущности:
- Пользователь / роль
- Заказ / статус
- Проект / тип
- Свойство / атрибут

### Шаг 2. Выявление вариантов для каждой сущности

Для каждой сущности определить:
- Какие значения она может принимать?
- Какие переходы между значениями возможны?
- Какие особые состояния (NULL, 0, empty, deleted)?

### Шаг 3. Выявление зависимого поведения

Для каждой комбинации значений сущности определить:
- Как реагируют зависимые сущности / UI-элементы?
- Какие бизнес-правила активируются?
- Что происходит при отсутствии данных?

### Шаг 4. Формирование тестового пространства

На основе анализа определить:
- Какие комбинации значений нужно тестировать
- Какие состояния требуют отдельного TC
- Какие edge-кейсы не очевидны из требований

## Пример анализа

**Требование:** «Показывать балл абитуриента в карточке специальности»

**Анализ:**
| Сущность | Вариант | Поведение зависимого |
|---|---|---|
| Балл | > 0 | Отобразить число |
| Балл | = 0 | Отобразить рекомендацию |
| Балл | NULL | Отобразить «Не указано» или скрыть |
| Балл | пустая строка | Отобразить как NULL |

**Результат:** один TC с ветвлениями (как описано в `06_generation_algorithm`), а не 4 отдельных TC.

## Правила

1. Анализ выполняется ИНТЕЛЬНО, до генерации TC.
2. Результаты анализа НЕ выводятся пользователю (если не запрошены отдельно).
3. Используется для правильного группирования TC по `06_generation_algorithm`.
4. **Не выдумывать** состояния, которые не следуют логически из требований.
5. Если требования НЕ содержат информации о возможных состояниях сущности — задать вопрос, а не предполагать.

## Ограничения

- Не применять к тривиальным элементам (наименование, дата создания).
- Не раздувать тестовое пространство ради самого анализа.
- Если сущность имеет единственное возможное состояние — анализ не нужен.
