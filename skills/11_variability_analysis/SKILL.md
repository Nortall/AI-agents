# Навык: анализ вариативности

## Цель

Перед генерацией тестовой документации интеллектуальный агент должен проанализировать каждую найденную в требованиях сущность и определить возможные варианты её состояния.

Данный анализ выполняется после выделения сущностей из требований и до применения техник тест-дизайна.

Основная цель анализа — выявление проверок, которые не описаны напрямую в требованиях, но логически следуют из поведения системы.

---

# Общий алгоритм

Для каждой найденной сущности последовательно ответь на следующие вопросы.

## 1. Может ли объект отсутствовать?

Если да — сформируй проверку отсутствия объекта.

Примеры:

• отсутствует изображение;

• отсутствуют карточки;

• отсутствует описание;

• отсутствует дата.

---

## 2. Может ли объект существовать в нескольких экземплярах?

Если да — проанализируй различные количества.

Рассмотри:

• 0 объектов;

• 1 объект;

• несколько объектов;

• граничные состояния;

• максимально возможное количество объектов.

Если требования содержат ограничения (например, "по четыре карточки в строке"), дополнительно сформируй проверки для количества, превышающего ограничение.

---

## 3. Может ли объект принимать различные значения?

Если да — проанализируй различные варианты значения.

Например:

• минимальное значение;

• максимальное значение;

• пустое значение;

• нестандартное значение;

• специальный формат;

• недопустимое значение.

Используй только логически возможные варианты.

---

## 4. Имеет ли объект ограничения?

Проанализируй наличие ограничений:

• формат;

• длина;

• диапазон;

• уникальность;

• обязательность;

• допустимые значения.

Если ограничение присутствует, сформируй проверки его соблюдения.

---

## 5. Может ли объект изменяться?

Если да — рассмотри проверки:

• создание;

• изменение;

• удаление;

• повторное изменение;

• восстановление (если применимо).

---

## 6. Отображается ли объект пользователю?

Если да — проанализируй возможные особенности отображения.

Например:

• корректность отображения;

• отображение при отсутствии данных;

• отображение длинных значений;

• отображение специальных символов;

• отображение различных размеров элементов;

• отображение различных форматов (например, файлы, изображения).

Не ограничивайся только требованиями к внешнему виду.

Оцени возможные пользовательские ситуации.

---

## 7. Влияет ли состояние объекта на другие элементы системы?

Если изменение объекта потенциально влияет на другие функции или сущности — **сгенерируй тестовые сценарии для проверки этого влияния.**

**Не пропускай этот шаг.** Если объект может быть изменён/удалён/изменён статус — проверь реакцию зависимых объектов.

### Направление: CRUD-цепочки (создание → изменение → удаление)

**Алгоритм анализа:**
1. Сущность может быть создана → значит, может быть удалена.
2. Сущность может быть изменена (статус, поля) → значит, зависимые сущности могут затронуться.
3. Найдём в требованиях зависимости (один объект → много объектов, «родитель → потомок», «список карточек к периоду», «специальности к УЗ»).
4. Для каждой зависимости сгенерируй TC: что происходит с дочерним объектом при изменении родителя.

**Пример 1: Период + Карточки**
Требование: «Пользователь может создать период и привязанные к нему карточки.»
- Период → CRUD (создание, изменение, удаление)
- Карточки → дочерний объект, зависящий от периода
→ TC: Удаление периода — проверить, что карточки удаляются/скрываются/остаются (реакция системы).
→ TC: Изменение периода — проверить, что карточки обновляются/сохраняются.

**Пример 2: Учебное заведение + Специальности**
Требование: «Реализовать страницу для просмотра информации по УЗ и его специальностям.»
- УЗ → CRUD (просмотр, изменение, потенциально удаление)
- Специальности → дочерний объект, зависимый от УЗ
→ TC: Удаление УЗ — проверить реакцию списка специальностей (удаляются/скрываются/остаются видимыми).
→ TC: Смена статуса УЗ (активно/неактивно) — проверить видимость специальностей.

**Пример 3: Категория товаров + Товары**
Требование: «Пользователь может создать категорию и товары внутри неё.»
- Категория → CRUD
- Товары → дочерний объект
→ TC: Удаление категории — товары в категории исчезают/перемещаются в «Без категории»/остаются.
→ TC: Переименование категории — названия товаров не меняются.

### Направление: Статусы и видимость

**Алгоритм анализа:**
1. Сущность имеет статус (активно/архив/удалено/на модерации)?
2. Изменение статуса → влияет на видимость/доступность дочерних объектов?
3. Сгенерируй TC на изменение статуса и проверку реакции зависимых объектов.

**Пример: Проект → Задачи**
Требование: «Менеджер может создать проект с задачами и менять статус проекта.»
- Проект → статус (активен, закрыт, архив)
- Задачи → дочерние объекты
→ TC: Закрытие проекта — задачи становятся недоступны для редактирования.
→ TC: Удаление проекта — задачи удаляются.

**Пример: Пользователь → Его записи**
Требование: «Пользователь может создавать записи и менять свой статус (активен/заблокирован).»
- Пользователь → статус (активен, заблокирован)
- Записи → дочерние объекты
→ TC: Блокировка пользователя — его записи становятся невидимы для других.
→ TC: Разблокировка пользователя — записи снова видимы.

### Направление: Связи между сущностями (много-ко-многим, привязки)

**Алгоритм анализа:**
1. Есть ли связь между двумя сущностями? (один к одному, один ко многим, много ко многим)
2. Что происходит при разрыве связи? (отвязка, удаление связи)
3. Что происходит при переназначении? (перенос к другому родителю)

**Пример: Пользователь → Роли**
Требование: «У пользователя могут быть назначены роли, менеджер может менять роли.»
- Пользователь → роли (много-ко-многим)
→ TC: Снятие всех ролей — пользователь теряет доступ.
→ TC: Смена роли — доступ обновляется.

### Важные правила

- **Не утверждай конкретное поведение** (не пиши «карточки будут удалены»). Пиши «проверить реакцию карточек при удалении периода».
- **Используй только логически обоснованные зависимости** — если сущность из требований не ссылается на другую, не придумывай связь.
- **Проверяй все CRUD-операции родителя**: создание (обычно описано в требованиях), изменение, удаление.
- **Если влияние невозможно логически предположить** (родительский объект не изменяется, не удаляется, не меняет статус) — пропускай этот шаг.

---

# Правила

Анализ выполняется для каждой найденной сущности.

Не использовать заранее подготовленные списки сущностей.

Работать исключительно через свойства объекта.

Не использовать знания конкретной предметной области.

---

# Пример

Требование:

"Карточка содержит изображение."

Ход анализа:

✓ Может отсутствовать?

Да.

→ Проверить отображение карточки без изображения.

✓ Может иметь различные размеры?

Да.

→ Проверить отображение маленького изображения.

→ Проверить отображение большого изображения.

→ Проверить отображение вертикального изображения.

→ Проверить отображение горизонтального изображения.

→ Проверить отображение разных форматов изображения.

✓ Влияет на другие элементы?

Нет.

---

Требование:

"Карточки выводятся по четыре в строке."

Ход анализа:

✓ Может быть различное количество карточек?

Да.

→ Проверить отображение одной карточки.

→ Проверить отображение двух карточек.

→ Проверить отображение четырёх карточек.

→ Проверить отображение пяти карточек.

→ Проверить отображение большого количества карточек.

---

# Ограничения

Если анализируемая ситуация не следует логически из требований или свойств объекта, не добавляй её.
