# Checklist Generation Rules

## Purpose

Чек-лист представляет собой компактный набор проверок, предназначенный для быстрого функционального тестирования.

В отличие от тестовых сценариев, чек-лист не содержит подробного описания действий, ожидаемых результатов, предусловий и тестовых данных.

Чек-лист используется:

- при регрессионном тестировании;
- для опытных QA-инженеров;
- при исследовательском тестировании;
- для проверки покрытия функциональности.

---

# General Principles

При генерации чек-листа необходимо:

- покрыть всю функциональность требований;
- учитывать позитивные и негативные сценарии;
- учитывать граничные случаи;
- учитывать проверки валидации;
- учитывать бизнес-правила;
- избегать дублирования;
- использовать краткие и понятные формулировки.

---

# Checklist Item Structure

Каждый пункт чек-листа представляет собой одну самостоятельную проверку.

Структура:

<что именно необходимо проверить>

или

Убедиться, что <ожидаемое поведение системы>

Примеры:

✔ Групповое создание классов

✔ Регистрация с уже существующим email.

✔ Валидация формата email.

✔ Минимальную длина пароля.

---

# Atomicity

Каждый пункт чек-листа проверяет только одну бизнес-идею.

Недопустимо:

Проверить регистрацию пользователя и авторизацию.

Правильно:

Проверить регистрацию пользователя.

Проверить авторизацию пользователя.

---

# Checklist Granularity

Чек-лист не должен превращаться в тест-кейс.

Не использовать:

- предусловия;
- шаги;
- ожидаемые результаты;
- тестовые данные;
- комментарии.

Каждый пункт должен быть максимально коротким.

---

# Coverage Rules

Чек-лист должен включать проверки:

- основной функциональности;
- альтернативных сценариев;
- негативных сценариев;
- граничных значений;
- валидации;
- обработки ошибок;
- ограничений бизнес-логики.

---

# Prioritization

Если требования позволяют определить критичность проверки, необходимо сгруппировать пункты по приоритетам.

Высокий

Средний

Низкий

Внутри каждой группы сначала располагаются наиболее критичные проверки.

---

# Duplicate Detection

Не допускается генерация нескольких пунктов, проверяющих одну и ту же бизнес-логику.

Перед добавлением нового пункта необходимо убедиться, что аналогичная проверка отсутствует.

---

# Hallucination Guard

Запрещается добавлять проверки, если соответствующее бизнес-правило отсутствует в требованиях.

Если проверка требует предположения о работе системы, она не должна включаться в итоговый чек-лист.

---

# Traceability

Каждый пункт чек-листа должен иметь источник происхождения в требованиях.

Если невозможно определить требование, которое проверяет данный пункт, такой пункт считается галлюцинацией и должен быть исключен.

---

# Formatting Rules

Использовать единый стиль оформления.


Не использовать:

- Возможно...
- Скорее всего...
- Вероятно...
- Если получится...

---

# Quality Criteria

Качественный чек-лист обладает следующими свойствами:

- покрывает все требования;
- не содержит дублирования;
- не содержит галлюцинаций;
- состоит из независимых проверок;
- имеет логичную последовательность;
- легко читается опытным QA-инженером.
