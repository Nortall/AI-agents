# Skill: Test IT Formatter

## Назначение

Данный skill отвечает исключительно за преобразование сгенерированной тестовой документации во внутренний формат системы Test IT.

Skill не принимает участия в анализе требований, проектировании тестов и определении их содержания.

Его задача — корректно оформить уже готовые тест-кейсы или чек-листы для последующего импорта в Test IT.

---

# Общие правила

После завершения генерации тестовой документации преобразуй результат в структуру, совместимую с импортом Test IT.

Не изменяй содержание тестов.

Не добавляй новые проверки.

Не удаляй существующие проверки.

Допускается только изменение структуры представления данных.

---

# Формат XLSX

Используй следующие колонки строго в указанном порядке.

1. Название
2. Описание
3. Статус
4. Приоритет
5. Действие
6. Ожидаемый результат
7. Действие предусловия
8. Ожидаемый результат предусловия
9. Действие постусловия
10. Ожидаемый результат постусловия
11. Продолжительность
12. Секция
13. Тестовые данные
14. Ревью
15. Вид

Изменение порядка колонок запрещено.

# Header-строка тест-кейса

Для каждой новой проверки создать Header-строку.

Обязательные поля:

Название

Описание

Статус

Приоритет

Продолжительность

Секция

Ревью

Вид

Остальные поля оставить пустыми.

---

## Правила заполнения Header

### Название

Использовать краткое и однозначное название тестового сценария.

---

### Описание

Использовать единый формат:

Источник: <путь требования>

Требование: <описание требования>

Например

Источник:
Solo Exhibition > Блоки для редактора > Заглавный блок

Требование:
Заглавный блок / Название

Не использовать расширения файлов.

Не писать ".md".

---

### Статус

Всегда:

Готов

---

### Приоритет

Использовать значение, определённое агентом генерации.

Допустимые значения:

Самый высокий

Высокий

Средний

Низкий

Самый низкий

---

### Продолжительность

Оценить ориентировочное время ручного прохождения.

Использовать формат

0h 10m 0s

---

### Секция

Использовать полный путь существующей папки Test IT.

Не изменять структуру секций.

Не создавать новые секции без явного указания.

---

### Ревью

Всегда

Не пройдено

---

### Вид

Допустимые значения

Позитивный

Негативный

Каждое действие должно иметь собственный ожидаемый результат.

Запрещается объединять несколько действий в одной строке.

Действие постусловия

Ожидаемый результат постусловия

# Правила оформления

Не заполнять поле

Тестовые данные

если это не требуется явно.

Не изменять текст тестовых сценариев.

Не сокращать шаги.

Не объединять несколько проверок.

Каждый тест-кейс должен экспортироваться независимо.

---

# Проверка структуры

Перед завершением формирования результата выполнить внутреннюю проверку.

Убедиться, что:

✓ порядок колонок соответствует шаблону;

✓ каждая Header-строка содержит обязательные поля;

✓ каждое действие имеет ожидаемый результат;

✓ каждое постусловие имеет ожидаемый результат;

✓ отсутствуют лишние секции;

✓ отсутствуют дубли тест-кейсов внутри одной секции.

# ПИТФАЛЫ при генерации XLSX (python + openpyxl)

## 1. Переменная `tc` = конфликт с параметром функции

**Никогда не используй `tc` как имя списка**, если в функции обработчике параметр тоже `tc`. Python не ругается, но функция читает список вместо параметра → `KeyError`.

Плохо:
```python
tc = [{"name": "TC-001", "duration": "0h 10m 0s", "section": "Поиск"}]

def write_header(tc):           # tc здесь — параметр, перекрывает список
    print(tc["duration"])       # KeyError!
```

Хорошо:
```python
cases = [{"name": "TC-001", "duration": "0h 10m 0s", "section": "Поиск"}]

def write_header(d):            # ✅ другое имя параметра
    print(d["duration"])        # Работает
```

## 2. Проверка ключей данных перед циклом

**Выведи `list(cases[0].keys())` перед циклом** и сравни с тем, что читает функция. Частая ошибка: данные имеют `"duration"`, `"section"`, а функция читает `"dur"`, `"sec"`.

## 3. openpyxl save delay

**После `wb.save()` подожди 1-2 секунды** перед проверкой `ls`. На macOS файл может быть создан, но не отобразился в FS. `exit 0` = успех.

## 4. f-string + фигурные скобки `{переменная}` в русском тексте

**F-строки Python интерпретируют `{любое_слово}` как переменную.** Если в тексте тестового описания (название, требование, шаг) есть фигурные скобки на русском — это сломает f-string с `NameError`.

Плохо:
```python
f"Требование: +{количество} по выбору"  # NameError: name 'количество' is not defined
```

Хорошо — экранировать через удвоение:
```python
f"Требование: +{{количество}} по выбору"  # → «+количество по выбору»
```

**Альтернатива:** использовать обычный `str` (без `f`) если в строке нет переменных — просто константа.

**Правило:** перед запуском `python3 generate_testit.py` всегда прогнать файл через `python3 -c "compile(open('generate_testit.py').read(), 'generate_testit.py', 'exec')"` — это ловит NameError ещё до `wb.save()`.

## 5. Рекомендуемый workflow генерации

1. Определи список с **полными ключами** (`name`, `description`, `status`, `priority`, `action`, `expected_result`, `precondition`, `postcondition`, `duration`, `section`, `review`, `type`)
2. Функции с **другими именами параметров** (`def write_header(d):`)
3. `print(list(data[0].keys()))` — проверь что все ключи на месте
4. Запусти цикл
5. `wb.save()` → `sleep(1)` → проверка файла
