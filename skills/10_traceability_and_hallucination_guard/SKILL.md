# Трассируемость и защита от галлюцинаций

## ЦЕЛЬ

Каждый тестовый сценарий должен быть явно связан с одним или несколькими требованиями.

Трассируемость используется для:
- контроля полноты покрытия требований
- обнаружения галлюцинаций
- выявления избыточных тестов
- проверки соответствия тестовых сценариев бизнес-требованиям

---

## ЗАПРЕЩЕНО ВЫДУМЫВАТЬ

Запрещено добавлять отсутствующие в требованиях сущности:
- статусы пользователей
- коды ошибок
- ограничения
- бизнес-правила
- атрибуты объектов

Если информации недостаточно, необходимо:
1. Сформировать предположение: `Предположение: «Минимальная длина пароля предполагается равной 8 символам.»`
2. Или сформировать вопрос: `«Уточните требования к сложности пароля.»`

Не создавать тестовые сценарии на основании неподтверждённых предположений.

---

## ДОПУСТИМЫЕ ИСТОЧНИКИ

Источником тестового сценария могут являться:
- основной сценарий
- альтернативный сценарий
- критерий приёмки
- явно сформулированное бизнес-правило
- нефункциональные требования (если они присутствуют)
- логические зависимости, выявленные через `11_variability_analysis` (см. ниже)

Не допускается создание тестовых сценариев без источника происхождения.

### Тесты из 11_variability_analysis — допустимы

Тестовые сценарии, выявленные через `11_variability_analysis` (анализ сущностей), считаются допустимыми, если:

1. Они логически следуют из **свойств объектов** в требованиях (сущность может быть создана → значит, может быть удалена).
2. Проверяют **реакцию зависимых объектов** на изменение родительского (удаление, изменение, скрытие).
3. Не выдумывают конкретное поведение системы (не утверждают «карточки удалятся» — проверяют «карточки реагируют»).

Пример:
Требование: «Пользователь может создать период с карточками.»
Через `11_variability_analysis`: период может быть удалён → карточки, привязанные к периоду, — зависимый объект.
→ TC допустим, источник: логическая зависимость «период → карточки».

Запрещено:
- Выдумывать новые сущности, которых нет в требованиях (например, «статус archived»).
- Приписывать системе конкретное поведение без подтверждения (например, «при удалении периода все карточки будут перемещены в корзину»).

---

## ТРЕБОВАНИЕ → ТЕСТЫ

Для каждого требования необходимо определить связанные тестовые сценарии.

Пример:
A1 Email уже существует → TC-002
A2 Неверный формат email → TC-003
A3 Слабый пароль → TC-004

Если требование не связано ни с одним тестовым сценарием, покрытие считается неполным.

---

## ТЕСТ → ТРЕБОВАНИЕ

Для каждого тестового сценария необходимо определить требование-источник.

Пример:
TC-001
Источник: Основной сценарий

TC-002
Источник: A1

TC-003
Источник: A2

TC-004
Источник: A3

Если источник определить невозможно, тестовый сценарий считается подозрительным и должен быть исключён.

---

## ОБНАРУЖЕНИЕ ГАЛЛЮЦИНАЦИЙ

Признаки галлюцинации:
- тест не связан ни с одним требованием
- тест использует несуществующие сущности
- тест использует отсутствующие статусы
- тест использует отсутствующие коды ошибок
- тест использует неописанные бизнес-правила

Пример галлюцинации:
Требование: «Пользователь регистрируется.»
Сгенерированный тест: «Проверка статуса pending_activation.»

Статус `pending_activation` отсутствует в требованиях. Такой тест — галлюцинация и должен быть удалён.

---

## ПРОВЕРКА ПОКРЫТИЯ

После генерации тестовых сценариев необходимо убедиться, что:
- основной сценарий покрыт тестами
- альтернативные сценарии покрыты тестами
- критерии приёмки покрыты тестами
- каждый тест имеет источник происхождения

---

## ДВУНАПРАВЛЕННАЯ МАТРИЦА ТРАССИРУЕМОСТИ

После генерации построить двунаправленную матрицу трассировки:

Требование → Тесты:
Основной сценарий → TC-001
A1 → TC-002
A2 → TC-003
Критерий приёмки AC-1 → TC-001

Тест → Требование:
TC-001 → Основной сценарий, AC-1
TC-002 → A1
TC-003 → A2

---

## ПРАВИЛА КАЧЕСТВА

НЕ допускается ситуация:
Одно требование → несколько тестов.

Пример нарушения:
A2 Невалидный email → TC-003 Отсутствует «@» → TC-004 Отсутствует домен → TC-005 Отсутствует локальная часть

Решение: объединить проверки валидации в один тест-кейс.

НЕ допускается ситуация:
Тестовый сценарий → отсутствие требования.
Такие тесты считаются галлюцинациями.

---

## АЛГОРИТМ САМОПРОВЕРКИ

После генерации выполнить:

1. Для каждого требования найти связанные тесты. Если тесты отсутствуют → создать дополнительные тестовые сценарии.
2. Для каждого тестового сценария найти источник происхождения. Если источник отсутствует → удалить тестовый сценарий.
3. Убедиться, что все критерии приёмки покрыты.
4. Вернуть только итоговый набор тестовых сценариев.

---

## ГЛАВНЫЙ ПРИНЦИП

Каждый тестовый сценарий должен отвечать на вопрос: «Какое именно требование я проверяю?»

Если ответ на этот вопрос отсутствует — тест не должен присутствовать в итоговом наборе.
