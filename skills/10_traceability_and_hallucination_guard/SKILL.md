# Трассируемость и защита от галлюцинаций

## Назначение

Каждый тестовый сценарий должен иметь явную связь с конкретным источником — требованием, критерием приёмки или бизнес-правилом из предоставленных материалов.

## Трассируемость (Traceability)

### Правило

Каждый TC должен иметь ссылку на источник, по которому он сгенерирован.

Формат:
```
Источник: <путь/название требования или критерий приёмки>
```

Пример:
```
TC-003 – Строка «Бюджет» — число при >0, рекомендация при 0/NULL, скрыта без бюджета

Приоритет: Высокий
Источник: Раздел 3.2 «Отображение бюджетной информации», критерий приёма 3.2.1

Предусловия:
...
```

### Матрица трассировки (внутренняя проверка)

Перед выводом результата внутренне проверь:
- Для каждого TC → существует ли требование-источник?
- Для каждого требования → есть ли хотя бы один TC, который его покрывает?

Не выводи матрицу пользователю, используй как механизм самопроверки.

## Защита от галлюцинаций (Hallucination Guard)

### Запрещено

- Выдумывать бизнес-правила, не указанные в требованиях.
- Придумывать статусы, коды ошибок, ограничения, API, формы данных, отсутствующие в требованиях.
- Добавлять функциональность, которую пользователь не запрашивал.
- Использовать примеры из других проектов как факт.

### Допустимо

- Логические выводы, вытекающие исключительно из предоставленных требований (например: если поле может быть пустым → проверить поведение с пустым значением).
- Использование `11_variability_analysis` для выявления потенциальных состояний сущностей (удаление, изменение статуса, отсутствие данных).

### Признак галлюцинации

Если TC проверяет поведение, которое:
- не упоминается в требованиях;
- не логически следует из требований;
- опирается на предположение «обычно так работает»;

→ это галлюцинация. Такой TC нужно удалить.

### Вопрос vs предположение

Если требования НЕ содержат достаточной информации:
- НЕ генерируй TC на основе догадок.
- Явно сформулируй вопрос для уточнения.

Формат вопроса:
```
❓ Вопрос: В требованиях не указано, что происходит при повторной отправке формы. Нужно ли тестировать дублирование записи?
```

## Проверка перед выводом

1. Для КАЖДОГО TC найди источник в требованиях.
2. Если источник не найден → TC удалить или запросить уточнение.
3. Для КАЖДОГО требования проверь, что хотя бы один TC его покрывает.
4. Проверь отсутствие внутренних API-кенов, названий БД и технических деталей, отсутствующих в требованиях.
